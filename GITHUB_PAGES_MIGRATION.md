# GitHub Pages deployment audit

This repo is ready to be deployed as a static site, but production verification still matters because the live site and the local repo have drifted before.

## What is already in place

- GitHub Pages workflow: `.github/workflows/deploy-github-pages.yml`
- Custom domain file: `CNAME`
- Jekyll bypass file: `.nojekyll`
- Build output goes to `public/`
- Static-safe contact flow: homepage forms open prefilled email drafts instead of relying on Netlify form handling
- Redirect support for clean paths such as:
  - `/affiliate-disclosure`
  - `/thanks`
  - `/tools/<slug>`
  - `/guides/<slug>`
  - `/comparisons/<slug>`

## Important reality check

The repo is deployment-ready, but that does not guarantee the live site is currently being served from this exact repo state.

Before a production release, verify all of the following:

1. The active production host
2. The active DNS target for `ihelpwithai.com`
3. Whether GitHub Pages is enabled for this repository
4. Whether the live site already matches the current branch tip

## Recommended deployment checklist

### If deploying through GitHub Pages

1. Confirm GitHub Pages is enabled for this repository and configured to use GitHub Actions.
2. Confirm `main` is the intended release branch.
3. Run `npm run build` locally and review the diff.
4. Push the desired commit to `main`.
5. Verify the GitHub Actions deployment succeeds.
6. Check the live site directly:
   - homepage
   - one tool page
   - one guide page
   - one comparison page
   - `best-free-ai-tools.html`
   - `editorial-methodology.html`

### If deploying through another static host

1. Run `npm run build`.
2. Deploy the `public/` directory.
3. Confirm the host preserves the custom domain and static redirects.
4. Re-run the same smoke test pages listed above.

## Current repo expectations

- Source files live at the repo root.
- `public/` is the deploy-ready mirror.
- Tool, guide, comparison, best-free, and methodology pages should be generated by the build script, not maintained by hand.
- Mailto forms are static-host compatible and are no longer a blocker for GitHub Pages.

## Production smoke test

After any release, verify:

- homepage search and filters load
- a tool page loads and prompt builder works and copies text
- a guide page loads
- a comparison page loads
- affiliate disclosure page loads
- best free tools page loads
- editorial methodology page loads
- `robots.txt` and `sitemap.xml` are reachable
